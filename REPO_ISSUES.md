# このリポジトリの課題点まとめ（2026-04-26時点）

## 1. ビルド・実行基盤の制約が強く、開発/CIの再現性が低い
- アプリは `net10.0-windows10.0.26100.1` をターゲットにした Windows 専用です。
- さらに WinForms と WPF を同時有効化し、`WMPLib` COM 参照も使っているため、クロスプラットフォームCIでの検証が難しい構成です。
- 実際、今回の検証環境では `dotnet` コマンド自体が無く、最低限のビルド確認もできませんでした。

## 2. プロジェクト設定の一貫性が弱く、保守時に混乱しやすい
- `LangVersion` が Debug/Release では `8.0`、別の PropertyGroup では `12.0` と重複定義されています。
- ブートストラッパーに `.NETFramework v4.8 / 3.5` の記述が残っており、現行ターゲット（.NET 10 + Windows）と混在しています。
- Debug の `OutputPath` が `..\`（リポジトリ直下）で、成果物の混入リスクがあります。

## 3. コードベースが巨大クラス中心で、責務分離が弱い
- `MainWindow.cs` が 2,700 行超、`AthleticView.cs` が 3,800 行超、`GUIDesigner.cs` が 2,900 行超と、単一ファイルに責務が集中しています。
- UI、描画、データ変換、I/O、状態管理が同居しやすく、変更時の影響範囲が広がりやすい状態です。

## 4. 例外処理の握りつぶしが散見され、障害解析しづらい
- `catch (Exception) {}` のような空ハンドリングが複数箇所にあり、失敗時に原因追跡しづらいです。
- 重要な JSON 変換やマップ読み込み処理でも握りつぶしがあるため、データ不整合が潜在化しやすくなります。

## 5. グローバル可変状態への依存が強い
- `Global` クラスに `MainWnd` / `definition` / `config` / `state` / `cpd` など可変の静的フィールドが集中しています。
- 依存関係が暗黙化し、単体テスト導入や将来の並列処理対応が難しくなります。

## 6. バイナリ同梱方針によるリポジトリ肥大化・更新コスト
- ルート配下に DLL や WebView2 セットアップ EXE などのバイナリが追跡されています。
- 差分レビュー性が低下し、セキュリティ/ライセンス確認・更新運用が重くなります。

## 7. 自動テスト基盤が見当たらず、回帰検知が手作業中心
- テストプロジェクト（unit/integration）が確認できず、品質担保が実機手動検証に寄りやすい構成です。

## 改善の優先順位（提案）
1. **可観測性の改善**：空 `catch` を段階的に削減し、ログ出力を追加。
2. **設定の整理**：`LangVersion`・ブートストラッパー・出力先の重複/旧設定を統一。
3. **分割統治**：巨大クラスを「UIイベント」「変換ロジック」「I/O」「描画」へ分離。
4. **テスト最小導入**：変換ロジック（HTML/JSON/マップ）から純粋関数化し、xUnit でスモークテスト追加。
5. **配布戦略見直し**：バイナリ直コミットを減らし、NuGet/リリース成果物へ寄せる。
